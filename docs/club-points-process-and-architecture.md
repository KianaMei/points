# 俱乐部员工积分系统文档索引

_本文件原先混合了流程和架构内容。为避免需求、流程、架构混在同一文档中，现改为索引文件。后续不要在这里继续追加业务细节。_

---

## 文档拆分

| 文档 | 职责 | 维护原则 |
| --- | --- | --- |
| `club-points-prd.md` | 产品需求、业务规则、验收标准、范围边界 | 只写需求，不写技术架构和数据库表设计 |
| `club-points-flow-design.md` | 流程图、状态机、关键异常分支 | 用 Mermaid 表达流程，作为流程源文件 |
| `club-points-module-design.md` | 模块拆分、模块边界、模块依赖、模块优先级 | 说明系统一共哪些模块，不写数据库表 |
| `club-points-architecture-design.md` | 架构方案、模块边界、事务边界、概念数据边界 | 不重复 PRD，不做最终物理数据库设计 |

## 推荐阅读顺序

1. 先读 `club-points-prd.md`，确认业务规则和范围。
2. 再读 `club-points-flow-design.md`，确认流程、状态机和异常分支。
3. 再读 `club-points-module-design.md`，确认系统模块拆分和模块边界。
4. 最后读 `club-points-architecture-design.md`，确认架构、事务边界和后续数据库设计约束。

## 下一步文档顺序

当前 `club-points-flow-design.md` 已包含主流程、异常分支、待办、通知、审计、并发、后台任务和状态机。后续如果继续细化，应优先补这些内容：

- 检查流程和 PRD 是否有矛盾。
- 按页面拆交互草图和字段清单。
- 按模块和用例拆接口清单。
- 补充后台任务运行参数、失败告警和人工处理页面。

流程稳定后，再进入数据库设计文档。

建议新增数据库设计文档：

- `club-points-database-design.md`

数据库设计必须基于 PRD、流程图和架构设计共同确认后再写，不能直接从页面表格或报表字段倒推表结构。
